# Cluster mTLS and membership ops

**Status:** Partial / disclosed ops guidance. This is **not** a production multi-host cluster claim and **not** a sharding design.

**Evidence:** `src/features/wdbx/cluster.zig`, `src/features/wdbx/cluster_rpc.zig`, `abi wdbx cluster serve`, `docs/spec/wdbx-north-star.mdx` §3.5, `docs/spec/non-loopback-rest-threat-review.mdx`.

## What exists today

| Control | Behavior |
| ------- | -------- |
| Transport | Real TCP RequestVote / AppendEntries (`cluster_rpc.zig`), loopback-tested |
| Auth | Shared secret frames via `ABI_WDBX_CLUSTER_TOKEN` (required for non-loopback binds) |
| Peers | Optional allowlist `ABI_WDBX_CLUSTER_PEERS` (comma-separated node ids) |
| TLS / mTLS | **Not linked in-process** — front with a TLS/mTLS-terminating proxy |
| Membership | Static config for the serve path; no dynamic join/leave protocol |
| Sharding | **Not implemented** (stays Proposed) |

## Threat model (routable cluster RPC)

1. **Token leak** grants Raft RPC participation / disruption on the bind.
2. **Cleartext frames** on the wire without an external TLS/mTLS front.
3. **Peer spoofing** when `ABI_WDBX_CLUSTER_PEERS` is unset on a routable bind.
4. **Availability** attacks against election/replication (DoS), even with a valid token.
5. **False production confidence** — loopback/routable demos are not multi-AZ ops.

## Decisions (claim-honest)

| Decision | Outcome |
| -------- | ------- |
| Native TLS/mTLS in ABI cluster RPC | **Deferred.** Prefer stunnel / nginx stream / Envoy / Caddy with mutual TLS to loopback `cluster serve` ports. |
| Dynamic membership | **Proposed.** Keep static TOKEN + PEERS until a tested join/leave path exists. |
| Data sharding | **Proposed / non-goal for this doc.** Do not schedule as “complete.” |
| Supported ops mode | Loopback multi-node demos and carefully controlled LAN binds behind mTLS proxy + token + peers. |

## Operator checklist

1. Prefer `127.0.0.1` for demos (`abi wdbx cluster serve <port>`).
2. For any non-loopback bind: set a strong `ABI_WDBX_CLUSTER_TOKEN`; set `ABI_WDBX_CLUSTER_PEERS`.
3. Terminate **mTLS** at the proxy; keep Zig listeners on loopback.
4. Do not claim production multi-host, HA, or sharding from this surface.
5. Pair with the REST/MCP HTTP review when the same host also exposes HTTP.

## Residual Proposed work

- Native TLS or mTLS linked and contract-tested on `cluster_rpc`.
- Dynamic membership / reconfiguration protocol with tests.
- Production multi-host runbooks (still not sharding).
