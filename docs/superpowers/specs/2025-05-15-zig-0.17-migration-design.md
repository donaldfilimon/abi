# Technical Roadmap: Migrating ABI to Zig 0.17 Master

## 1. Migration Strategy: Feature-Branch vs. Keep 0.16
*   **Assessment**: The ABI framework relies heavily on custom build-time linking workarounds and specific Zig 0.16 primitives. A direct migration of the entire codebase is high-risk due to the potential for subtle changes in the 0.17 self-hosted compiler and the fragile nature of our current macOS linker workarounds.
*   **Recommendation**: **Feature-branch migration**.
    *   Maintain `main` on 0.16 for stability and active production support.
    *   Create `mig/zig-0.17` branch to house the migration.
    *   Goal: Verify the core ABI framework + inference engine compiles and passes tests on 0.17 before merging.

## 2. Phase-by-Phase Migration Plan

### Phase 1: Infrastructure & Build System (High Priority)
*   **Goal**: Ensure `zig build` works in 0.17.
*   **Actions**:
    *   **Build API (`build.zig`)**: 0.17 introduces stricter `std.Build` ergonomics. Replace `b.addLibrary` (which is legacy/removed) with the new `b.addStaticLibrary` pattern.
    *   **Module API**: Verify `b.addModule` vs `b.createModule`. The module management has changed; ensure `abi` imports are correctly resolved.
    *   **Custom Linker (`build/linking.zig`)**: 0.17's internal Mach-O linker has improved. Test if `artifact.use_lld = false` is still strictly necessary or if we can use `--disable-lld` or other 0.17 linker flags.
    *   **Compiler Overrides**: Check if `build.sh`'s `is_macos_26_4_or_newer` logic needs to be updated to detect 0.17 pathing changes.

### Phase 2: Standard Library & Language Primitives (Medium Priority)
*   **Goal**: Fix build errors due to stdlib deprecations.
*   **Actions**:
    *   **`ArrayList`**: Replace legacy `init` patterns with 0.17 compliant initializers.
    *   **String Manipulation**: Replace `trimRight` with `trimEnd`.
    *   **Time**: Replace `milliTimestamp` with `std.time.milliTimestamp` (if renamed/moved) or equivalent foundation wrappers.
    *   **Error Sets**: 0.17 has stricter error set inference; review `Foundation.errors` and `src/core/errors.zig`.

### Phase 3: Validation & Parity Gate (Critical)
*   **Goal**: Verify implementation matches stubs.
*   **Actions**:
    *   Run `zig build check-parity` to ensure feature parity.
    *   Address all `std.testing` failures, especially in `src/inference/` where connector tests are known fragile.
    *   Update CI scripts (`tools/ci/`) to target the 0.17 build.

## 3. Detailed Technical Tasks
1.  **Update `build.zig`**:
    *   Transition `b.addLibrary` to `b.addStaticLibrary`.
    *   Validate `Module` creation and import mapping.
2.  **Adapt macOS Linker Workarounds**:
    *   Investigate `use_lld = false` impact in 0.17.
    *   Check `artifact.root_module.resolved_target` changes for 0.17 compiler internals.
3.  **Address Deprecations**:
    *   Bulk-find `trimRight` and update to `trimEnd`.
    *   Review `ArrayList` usage across `src/features/` and `src/foundation/`.
    *   Apply `foundation.time` updates.

## 4. Migration Effort
*   **Level**: High.
*   **Justification**: The framework's core relies on low-level system integration (linker patching) that is highly sensitive to Zig's compiler/linker evolution.
*   **Workload**: ~2-3 developer-days for base migration; +1 day for testing inference engine stability.
