# ABI Architecture & Module Guidelines

## Framework Overview

ABI is a modular Zig 0.16 framework (pinned `0.16.0-dev.2962+08416b44f`, package version `0.4.0`) for distributed AI systems, vector search, and GPU compute. It uses comptime feature gates controlled via `build.zig` flags.

## Named Modules (dev.2962+)

Only 3 named modules exist:

| Module | Root Source | Purpose |
|--------|-----------|---------|
| `abi` | `src/root.zig` | **All** source files under `src/` belong to this single module |
| `build_options` | (synthetic) | Comptime feature flags generated by `build/modules.zig` |
| `cli` | `tools/cli/mod.zig` | CLI tool module (imports `abi`) |

The `foundation` namespace (`abi.foundation`) is part of the `abi` module, accessed via `src/services/shared/mod.zig`. It is **not** a separate named module.

**Critical rule**: Every `.zig` file belongs to exactly one module. All `src/` files are in the `abi` module. There are NO `shared_services`, `core`, or `wdbx` named modules.

├── features/                   # Comptime-gated domain implementations (19 dirs)

## Import Rules

- **External consumers** (CLI, tests, examples): `@import("abi")`
- **Within `src/`**: Use relative paths with explicit `.zig` extensions:
  ```zig
  const config = @import("../config/mod.zig");         // relative within src/
  const simd = @import("../../services/shared/simd/mod.zig");  // cross-zone OK (same module)
  ```
- **Foundation namespace**: Access via `@import("abi").foundation` (external) or relative `@import("../../services/shared/mod.zig")` (internal) — no separate named module
- **Feature flags**: `@import("build_options")` (named module, available everywhere)
- **NEVER** use `@import("shared_services")` or `@import("core")` — these named modules no longer exist
- Enforced by `zig build check-imports`

## Feature Flag Conventions

- Prefix: `feat_<name>` (NOT `enable_<name>`), all default to `true` (except `feat_mobile`)
- 27 flags in `build/options.zig` (including 2 internal: `feat_explore`, `feat_vision`)
- 58 combos validated in `build/flags.zig`
- Flags exported via `build_options` module
- Disable: `zig build -Dfeat-gpu=false`
- New in dev.2962: `feat_lsp`, `feat_mcp` flags for service gating

## mod/stub Contract

Every comptime-gated feature has paired files:
- `mod.zig` — real implementation (compiled when feature enabled)
- `stub.zig` — empty implementation (compiled when feature disabled)

**Stubs must match mod public signatures exactly.** The compiler enforces this via dual imports in `src/core/framework/feature_imports.zig`.

|-------------|-----------|---------|

## Build System

- `build/modules.zig` — Module creation (`wireAbiImports(module, build_opts)` wires `build_options`; no separate foundation module)
- `build/module_catalog.zig` — Feature module registration, catalog metadata, and 179 feature test entries
- `build/options.zig` — Feature flag definitions (27 flags in `CanonicalFlags`)
- `build/flags.zig` — Validation matrix (58 flag combos)
- `build/test_discovery.zig` — Unified `abi` module as test root (single-module ownership; previous per-entry approach removed)
- `build/link.zig` — Platform-specific linking (macOS frameworks, Linux/Windows/BSD/Android/illumos/Haiku libs)
- `build/targets.zig` — 35 example targets + 18 cross-compilation targets
- `build/cli_tests.zig` — CLI test steps: `cli-tests` (smoke, ~53 vectors) and `cli-tests-full` (exhaustive)

## Performance & SIMD

- Use `services/shared/simd/mod.zig` for portable hardware acceleration
- Prefer `std.Io` patterns for non-blocking I/O
- Target `native` by default, validate for `wasm32-freestanding`
